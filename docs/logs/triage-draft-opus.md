# Security Triage Verdict Draft (Next.js Static Export)

**Timestamp (UTC):** 2026-03-19 03:05:30Z

This triage evaluates 18 audit findings against the verified architecture and data-flow reality of this project: a Next.js `output:"export"` static site on Vercel with no runtime server logic and no user-input paths. Severity and validity were reassessed using practical exploitability, required attacker capabilities, and actual impact in this threat model.

| ID | Original | Verdict | Corrected Severity | Justification |
|---|---|---|---|---|
| V-1 | High | DOWNGRADE (Low) | Low | The site is statically exported (`output:"export"`) and served without a Node.js runtime, which sharply reduces header-related exploit paths compared with dynamic apps. Missing explicit CSP/frame/content-type hardening is still a defense-in-depth gap, but with no user input and developer-authored static content, practical risk is limited. Vercel defaults further reduce baseline exposure, so Low is appropriate. |
| X-1 | Medium | FALSE POSITIVE | None (Not a vulnerability) | `rehype-raw` plus `dangerouslySetInnerHTML` is exploitable when untrusted input reaches rendering, but Facts #4, #5, and #7 confirm no user input and only build-time, repo-authored markdown. Content is imported from static `docs.json`, not runtime-submitted data. An attacker would need repo commit/build pipeline control, which already implies total compromise beyond XSS. |

## Summary

Of 18 findings, **1 is VALID (Info)**, **6 are DOWNGRADED** (to Low/Info), and **11 are FALSE POSITIVE / not applicable** under the confirmed static-site threat model. The main actionable security item is **V-1** (host-level security headers, especially CSP) as defense-in-depth. Most other items are either architecture notes, quality/hygiene concerns, or findings that assume user input/server runtime paths that do not exist in this deployment.
