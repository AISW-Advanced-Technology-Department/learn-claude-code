## Executive Summary

The application is a **static documentation site** with no user input processing, no backend APIs, and no runtime server-side code. All content is sourced from build-time generated JSON files derived from trusted repository files.

**All 7 flagged findings are SAFE PATTERNS** — none represent exploitable runtime vulnerabilities in the current architecture. However, **Finding #1 represents a latent supply-chain XSS risk** that warrants defense-in-depth mitigation.

Two additional findings were identified: missing security headers and missing `rel="noreferrer"` on an external link.

## Finding Analysis

### FINDING 1: `dangerouslySetInnerHTML` in doc-renderer.tsx

| Attribute | Value |
|---|---|
| **File** | `web/src/components/docs/doc-renderer.tsx:87` |
| **Classification** | ⚠️ **SAFE PATTERN** (with supply-chain caveat) |
| **Severity** | **Low** (Info for runtime; Low-Medium if build pipeline compromised) |
| **CVSS 3.1** | 0.0 runtime / ~3.7 supply-chain |
| **Reviewer Consensus** | Both reviewers agree |

**Code:**
```tsx
dangerouslySetInnerHTML={{ __html: html }}
```

**Data Flow:**
1. `docsData` ← static import from `@/data/generated/docs.json` (build-time artifact)
2. `docs.json` ← generated by `scripts/extract-content.ts` from repo's `docs/` markdown files
3. Markdown pipeline: `remarkParse → remarkGfm → remarkRehype({allowDangerousHtml:true}) → rehypeRaw → rehypeHighlight → rehypeStringify`
4. Post-processed by `postProcessHtml()` (regex-based class additions — no sanitization)
5. Rendered via `dangerouslySetInnerHTML`

**Why SAFE at runtime:** No user input reaches this pipeline. All content is from trusted repository files, compiled at build time. The attack surface requires repository write access.

**Why flagged:** The pipeline explicitly enables raw HTML pass-through (`allowDangerousHtml: true` + `rehypeRaw`) with **zero sanitization**. A malicious contributor could submit a PR with `<img onerror="...">` or `<script>` in a markdown doc. If it passes code review, the payload executes in every visitor's browser.

**postProcessHtml() note:** Reviewed by both analysts — all regex patterns are static literals. No regex injection possible, but no security-relevant filtering either; malicious raw HTML passes through unchanged.

**Recommendations:**
- **HIGH PRIORITY**: Add `rehype-sanitize` to the unified pipeline before `rehype-stringify`. Use GitHub's allowlist schema.
- Consider removing `allowDangerousHtml: true` if raw HTML in docs is not required.

### FINDING 2: `dangerouslySetInnerHTML` in layout.tsx (Theme Script)

| Attribute | Value |
|---|---|
| **File** | `web/src/app/[locale]/layout.tsx:41-48` |
| **Classification** | ✅ **SAFE PATTERN** |
| **Severity** | **Info** |
| **CVSS 3.1** | 0.0 |
| **Reviewer Consensus** | Both reviewers agree |

**Code:**
```tsx
<script dangerouslySetInnerHTML={{ __html: `
  (function() {
    var theme = localStorage.getItem('theme');
    if (theme === 'dark' || (!theme && window.matchMedia('(prefers-color-scheme: dark)').matches)) {
      document.documentElement.classList.add('dark');
    }
  })();
`}} />
```

**Analysis:** Standard FOUC prevention pattern. The template literal contains **zero interpolated variables**. The `localStorage.getItem('theme')` value is only compared to the literal string `'dark'` — never inserted into DOM, never passed to `eval()`. Even with localStorage poisoning, there is no exploitable code path.

**Recommendations:**
- **LOW PRIORITY**: If strict CSP is adopted, convert to nonce/hash-based script or external file.

### FINDING 5: `window.location.href` Assignment in header.tsx

| Attribute | Value |
|---|---|
| **File** | `web/src/components/layout/header.tsx:45` |
| **Classification** | ✅ **SAFE PATTERN** |
| **Severity** | **Info** |
| **CVSS 3.1** | 0.0 |
| **Reviewer Consensus** | Both reviewers agree |

**Code:**
```ts
function switchLocale(newLocale: string) {
  const newPath = pathname.replace(`/${locale}`, `/${newLocale}`);
  window.location.href = newPath;
}
```

**Analysis:**
- `newLocale` sourced from hardcoded array `["en", "zh", "ja"]` via `onClick` handlers
- `pathname` from Next.js `usePathname()` (always a relative path, no protocol)
- `pathname.replace()` performs string substitution between known locale prefixes
- Not callable with arbitrary URLs

**Open Redirect check:** `window.location.href` with a relative path (no protocol prefix) is same-origin navigation. `usePathname()` cannot return protocol-prefixed values. Not exploitable.

**Recommendations:**
- **LOW PRIORITY**: Consider using `next/navigation` router for client-side navigation.

### FINDING 6: Type Coercion with `as any` in i18n.tsx

| Attribute | Value |
|---|---|
| **File** | `web/src/lib/i18n.tsx:28-30` |
| **Classification** | ✅ **SAFE PATTERN** (Code Quality Issue) |
| **Severity** | **Info** |
| **CVSS 3.1** | 0.0 |
| **Reviewer Consensus** | Both reviewers agree |

**Code:**
```tsx
const ns = namespace ? (messages as any)[namespace] : messages;
if (!ns) return key;
return (ns as any)[key] || key;
```

**Analysis:** `messages` sourced from statically imported JSON. `namespace` and `key` are developer-provided string literals. No runtime user input.

**Prototype pollution check:** Accessing `(messages as any)["__proto__"]` or `["constructor"]` returns `Object.prototype` or `Object`, but the result is only read for a translation value — never assigned to or used for mutation. The `|| key` fallback ensures safe degradation. **Not exploitable.**

**Recommendations:**
- **LOW PRIORITY**: Replace `as any` with proper TypeScript typing.

## Additional Findings

### ADDITIONAL-1: Missing Security Headers

| Attribute | Value |
|---|---|
| **Classification** | ⚠️ **MISSING CONTROL** |
| **Severity** | **Medium** |
| **Reviewer Consensus** | Both reviewers agree |

Neither `next.config.ts` nor `vercel.json` configures security response headers:
- `Content-Security-Policy`
- `X-Content-Type-Options`
- `X-Frame-Options`
- `Referrer-Policy`
- `Permissions-Policy`
- `Strict-Transport-Security`

Absence of CSP means any future XSS (e.g., via Finding #1's supply-chain vector) would have unrestricted script execution.

**Recommendation (HIGH PRIORITY):** Add security headers to `vercel.json`:
```json
{
  "headers": [
    {
      "source": "/(.*)",
      "headers": [
        { "key": "X-Content-Type-Options", "value": "nosniff" },
        { "key": "X-Frame-Options", "value": "DENY" },
        { "key": "Referrer-Policy", "value": "strict-origin-when-cross-origin" },
        { "key": "Permissions-Policy", "value": "camera=(), microphone=(), geolocation=()" },
        { "key": "Content-Security-Policy", "value": "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; font-src 'self'; frame-ancestors 'none'; base-uri 'self'; form-action 'self'" }
      ]
    }
  ]
}
```

**Peer Reviewer Note:** Long-term, avoid reliance on `'unsafe-inline'` in CSP. Use nonces or hashes for the theme script.

### ADDITIONAL-2: Missing `rel="noreferrer"` on External Link

| Attribute | Value |
|---|---|
| **Classification** | ✅ **SAFE PATTERN** (Minor Hardening) |
| **Severity** | **Info** |

External GitHub link in `header.tsx` uses `rel="noopener"` but not `rel="noreferrer"`. Modern browsers handle `target="_blank"` safely with just `noopener`.

**Recommendation:** Add `noreferrer`: `rel="noopener noreferrer"`.

## Overall Risk Assessment

This is a **low-risk application** by design. The static-export architecture eliminates entire categories of web vulnerabilities (SSRF, SQLi, command injection, authentication bypass, IDOR). The primary residual risk is **supply-chain integrity**:

1. **Repository contributors** — malicious PRs with XSS in markdown docs
2. **npm dependency chain** — especially unified/remark/rehype ecosystem
3. **CI/CD pipeline** — build artifact integrity

The recommended mitigations (sanitization + CSP + dependency monitoring) provide effective defense-in-depth.
