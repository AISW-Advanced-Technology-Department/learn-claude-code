## Executive Summary

The application is a **static documentation site** with no user input processing, no backend APIs, and no runtime server-side code. All content is sourced from build-time generated JSON files derived from trusted repository files.

**All 7 flagged findings are SAFE PATTERNS** — none represent exploitable runtime vulnerabilities in the current architecture. However, **one finding (Finding #1) represents a latent supply-chain XSS risk** that both reviewers recommend mitigating with defense-in-depth measures.

The primary security concern is **the absence of security headers** (CSP, X-Frame-Options, etc.) and the **unsanitized HTML rendering pipeline** that could amplify impact if the build pipeline or repository is ever compromised.

## Finding Analysis

### FINDING 1: `dangerouslySetInnerHTML` in doc-renderer.tsx

| Attribute | Value |
|---|---|
| **File** | `web/src/components/docs/doc-renderer.tsx:87` |
| **Classification** | ⚠️ **SAFE PATTERN** (with supply-chain caveat) |
| **Severity** | **Low** (Info for runtime; Low-Medium if build pipeline compromised) |
| **CVSS 3.1** | 0.0 runtime / 3.7 supply-chain (AV:N/AC:H/PR:H/UI:R/S:U/C:L/I:L/A:N) |

**Code:**
```tsx
dangerouslySetInnerHTML={{ __html: html }}
```

**Data Flow Analysis:**
1. `docsData` ← static import from `@/data/generated/docs.json` (build-time artifact)
2. `docs.json` ← generated by `scripts/extract-content.ts` from repo's `docs/` markdown files
3. Markdown processed through: `remarkParse → remarkGfm → remarkRehype({allowDangerousHtml:true}) → rehypeRaw → rehypeHighlight → rehypeStringify`
4. HTML post-processed by `postProcessHtml()` (regex-based class additions, no sanitization)
5. Final HTML rendered via `dangerouslySetInnerHTML`

**Why SAFE at runtime:** No user input reaches this pipeline. All content is from trusted repository files, compiled at build time into a static JSON bundle. The attack surface requires repository write access.

**Why flagged with caveat:** The pipeline explicitly enables raw HTML pass-through (`allowDangerousHtml: true` + `rehypeRaw`) and performs **zero sanitization**. If a malicious contributor submits a PR with `<img onerror="...">` or `<script>` tags in a markdown doc file, and it passes code review, the payload would execute in every visitor's browser.

**Recommendations:**
1. **HIGH PRIORITY**: Add `rehype-sanitize` to the unified pipeline before `rehype-stringify`. Use a strict allowlist schema (e.g., GitHub's schema).
2. Consider stripping `allowDangerousHtml: true` if raw HTML in docs is not required.

### FINDING 2: `dangerouslySetInnerHTML` in layout.tsx (Theme Script)

| Attribute | Value |
|---|---|
| **File** | `web/src/app/[locale]/layout.tsx:41-48` |
| **Classification** | ✅ **SAFE PATTERN** |
| **Severity** | **Info** |
| **CVSS 3.1** | 0.0 |

**Code:**
```tsx
<script dangerouslySetInnerHTML={{ __html: `
  (function() {
    var theme = localStorage.getItem('theme');
    if (theme === 'dark' || (!theme && window.matchMedia('(prefers-color-scheme: dark)').matches)) {
      document.documentElement.classList.add('dark');
    }
  })();
`}} />
```

**Analysis:** This is a standard FOUC (Flash of Unstyled Content) prevention pattern. The template literal contains **zero interpolated variables** — it is a fully hardcoded string. The `localStorage.getItem('theme')` call reads a value but only compares it to the literal string `'dark'`; the value is never inserted into DOM or used in `eval()`.

**Security note on localStorage poisoning:** Even if an attacker could set `localStorage.theme` to an arbitrary value (e.g., via another XSS on the same origin), this script only checks `=== 'dark'` and calls `classList.add('dark')`. There is no code path where the stored value is reflected into the DOM.

**Recommendations:**
- **LOW PRIORITY**: If strict CSP is adopted, replace with a nonce-based or hash-based script, or extract to an external JS file.

| Attribute | Value |
|---|---|
| **File** | `web/scripts/extract-content.ts:88` |
| **Classification** | ✅ **SAFE PATTERN** (False Positive) |
| **Severity** | **Info** |
| **CVSS 3.1** | 0.0 |

**Code:**
```ts
const toolPattern = /"name"\s*:\s*"(\w+)"/g;
while ((m = toolPattern.exec(source)) !== null) { ... }
```

**Analysis:** This is `RegExp.prototype.exec()`, a standard JavaScript regex matching method. It is **not** `child_process.exec()`. The script runs only at build time (`npm run extract`), never in the browser. The regex pattern `/"name"\s*:\s*"(\w+)"/g` is bounded by `\w+` (alphanumeric only) and is not susceptible to ReDoS.

**Recommendations:** None required.

### FINDING 5: `window.location.href` Assignment in header.tsx

| Attribute | Value |
|---|---|
| **File** | `web/src/components/layout/header.tsx:45` |
| **Classification** | ✅ **SAFE PATTERN** |
| **Severity** | **Info** |
| **CVSS 3.1** | 0.0 |

**Code:**
```ts
function switchLocale(newLocale: string) {
  const newPath = pathname.replace(`/${locale}`, `/${newLocale}`);
  window.location.href = newPath;
}
```

**Analysis (Open Redirect assessment):**
- `newLocale` is sourced from `onClick` handlers bound to buttons generated from a **hardcoded** array: `["en", "zh", "ja"]`
- `pathname` comes from Next.js `usePathname()` which returns the current page's path
- `pathname.replace()` performs a string substitution — replacing one known locale prefix with another
- The function is **not callable with arbitrary URLs**; it is an internal locale switcher

**Attack vector check:** Could a crafted pathname like `/en/javascript:alert(1)` cause issues? No — `window.location.href` with a relative path (no protocol) is interpreted as a same-origin navigation. The `pathname` from `usePathname()` cannot contain protocol prefixes.

**Recommendations:**
- **LOW PRIORITY**: Consider using Next.js `useRouter().push()` for client-side navigation instead of full page reload.

### FINDING 6: Type Coercion with `as any` in i18n.tsx

| Attribute | Value |
|---|---|
| **File** | `web/src/lib/i18n.tsx:28-30` |
| **Classification** | ✅ **SAFE PATTERN** (Code Quality Issue) |
| **Severity** | **Info** |
| **CVSS 3.1** | 0.0 |

**Code:**
```tsx
const ns = namespace ? (messages as any)[namespace] : messages;
if (!ns) return key;
return (ns as any)[key] || key;
```

**Analysis:** `messages` is sourced from statically imported JSON files (`en.json`, `zh.json`, `ja.json`). `namespace` and `key` are developer-provided string literals in component code (e.g., `useTranslations("nav")` → `t("timeline")`). There is no runtime user input flowing into these parameters.

**Prototype pollution check:** Accessing `(messages as any)[namespace]` with `namespace = "__proto__"` or `"constructor"` would return `Object.prototype` or `Object`, but the result is only used for reading a translation string — it is never assigned to or used for object mutation. The fallback `|| key` ensures graceful degradation. Not exploitable.

**Recommendations:**
- **LOW PRIORITY**: Replace `as any` with proper TypeScript typing using `keyof` for better maintainability.

## Additional Findings

### ADDITIONAL-1: Missing Security Headers

| Attribute | Value |
|---|---|
| **Classification** | ⚠️ **MISSING CONTROL** |
| **Severity** | **Medium** |

**Analysis:** Neither `next.config.ts` nor `vercel.json` configures any security response headers. The site is served without:
- `Content-Security-Policy`
- `X-Content-Type-Options`
- `X-Frame-Options`
- `Referrer-Policy`
- `Permissions-Policy`
- `Strict-Transport-Security`

For a static site, the impact is reduced, but absence of CSP means any future XSS introduction (e.g., via Finding #1's supply-chain vector) would have unrestricted script execution.

**Recommendation (HIGH PRIORITY):** Add to `vercel.json`:
```json
{
  "headers": [
    {
      "source": "/(.*)",
      "headers": [
        { "key": "X-Content-Type-Options", "value": "nosniff" },
        { "key": "X-Frame-Options", "value": "DENY" },
        { "key": "Referrer-Policy", "value": "strict-origin-when-cross-origin" },
        { "key": "Permissions-Policy", "value": "camera=(), microphone=(), geolocation=()" },
        { "key": "Content-Security-Policy", "value": "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; font-src 'self'; frame-ancestors 'none'; base-uri 'self'; form-action 'self'" }
      ]
    }
  ]
}
```

### ADDITIONAL-2: Missing `rel="noreferrer"` on External Links

| Attribute | Value |
|---|---|
| **Classification** | ✅ **SAFE PATTERN** (Minor) |
| **Severity** | **Info** |

**Analysis:** External GitHub link at `header.tsx:98` uses `rel="noopener"` but not `rel="noreferrer"`. Modern browsers handle `target="_blank"` safely with just `noopener`, and Next.js adds `rel="noopener noreferrer"` automatically for its `Link` component. The raw `<a>` tag omits `noreferrer`.

**Recommendation:** Add `noreferrer` for completeness: `rel="noopener noreferrer"`.

## Overall Risk Assessment

This is a **low-risk application** by design. The static-export architecture eliminates entire categories of web vulnerabilities (SSRF, SQLi, command injection, authentication bypass, IDOR). The primary residual risk is **supply-chain integrity** — the trust boundary extends to:
1. Repository contributors (malicious PRs with XSS in markdown docs)
2. npm dependency chain (especially unified/remark/rehype ecosystem)
3. CI/CD pipeline (build artifact integrity)

The recommended mitigations (sanitization + CSP + dependency monitoring) provide effective defense-in-depth against these supply-chain vectors.
