## 2. レビュー結果

### 2.1 レポートの分析は適切か？見落としている脅威はないか？

**評価: 概ね適切**

`security-audit-report.md` は、Next.js の静的エクスポートという特性を正しく理解し、攻撃面がサーバーサイド（SQLインジェクション、RCE等）ではなくクライアントサイド（XSS、DOM Injection）に集中していることを適切に評価しています。

- **評価できる点:**
    - `dangerouslySetInnerHTML` のリスクを「Markdownがリポジトリ管理下にあるため条件付きで許容範囲」としている点は現実的かつ妥当です。
    - CSP (Content Security Policy) の欠如を最大のリスクとして指摘している点は、静的サイトの防御策として非常に的確です。

- **インフラ視点での補足（見落としというより範囲外）:**
    - **Basic認証の運用リスク:** アプリケーション診断の範囲外ですが、全員共通のID/パスワードを用いることによる「退職者リスク（パスワード変更の運用コスト）」や「ログ監査の不可能性（誰がアクセスしたか不明）」についての言及が必要です。
    - **Nginx設定:** 静的ファイルを配信するNginxの堅牢化（バージョン隠蔽、不要なメソッドの拒否など）についての言及が必要です。

### 2.2 社内イントラネットでの静的サイトホスティングという用途に対してリスク評価は妥当か？

**評価: 妥当**

外部公開しないため、不特定多数からの攻撃リスクは極めて低いです。レポートが指摘する「内部犯行やサプライチェーン攻撃（悪意あるMarkdownのコミット）」が主な脅威シナリオとなる点も同意します。

- **追加のリスク評価:**
    - **ラテラルムーブメント:** 万が一、イントラネット内の他の端末がマルウェアに感染した場合、ポート8314へのアクセスが可能になります。Basic認証はこの防御層として機能しますが、共通パスワードが漏洩していると突破されます。ただし、静的サイトであるため、侵入されてもサーバー側の情報を盗むことはできず（ファイル閲覧のみ）、サーバーを踏み台にすることも困難（静的ファイル配信のみのため）であり、リスク受容可能です。

### 2.3 Docker化する際に追加で注意すべきセキュリティ事項はあるか？

`web/Dockerfile` が現状存在しないため、新規作成および構築時の推奨事項を以下に挙げます。

1.  **Nginxの堅牢化:**
    - `server_tokens off;` を設定し、Nginxのバージョン情報を隠蔽する。
    - 不要なHTTPメソッド（POST, PUT, DELETE等）を拒否する設定を入れる。
    - 適切なタイムアウトとバッファサイズを設定し、DoS耐性を高める。
2.  **コンテナの非root実行:**
    - Dockerコンテナ内では `root` ではなく `nginx` ユーザー（または専用の非特権ユーザー）でプロセスを実行するように構成する（ポート1024以上を使用するなど）。
3.  **Basic認証ファイルの管理:**
    - `.htpasswd` ファイルをDockerイメージ内に焼き込まないこと（リポジトリにコミットしない）。
    - 起動時に環境変数から生成するか、Kubernetes Secrets/Docker Secrets等を用いてボリュームマウントする方式を採用する。
4.  **セキュリティヘッダーの付与 (Nginx設定):**
    - アプリケーションレポートで指摘されているCSPやセキュリティヘッダーは、Next.js側 (`next.config.js`) ではなく、Nginxの設定 (`add_header`) で付与するのが確実かつ効率的です。
    - 推奨ヘッダー: `Content-Security-Policy`, `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`

### 2.4 next@16.1.6の脆弱性3件は静的エクスポートでも影響があるか？

**結論: 影響なし**

`dependency-scan.md` で検出された3件の脆弱性は、いずれも静的エクスポート (`output: "export"`) 環境下では動作しない機能に関するものです。

1.  **GHSA-mq59-m269-xvcx (Server Actions CSRF):** 静的エクスポートでは Server Actions は無効化されるため影響しません。
2.  **GHSA-3x4c-7xq6-9pq8 (Image Optimization Cache):** `next start` でサーバーを起動する場合の影響です。静的エクスポートおよび `images: { unoptimized: true }` 設定下では、Next.js の画像最適化APIは動作しないため影響しません。
3.  **GHSA-ggv3-7p47-pfv8 (Request Smuggling via Rewrites):** Next.js の Node.js サーバー機能に依存する脆弱性です。Nginx配信環境では影響しません。

ただし、将来的にSSR/ISR機能を利用する可能性に備え、あるいは脆弱性スキャナーのアラートを減らすために、開発サイクルの中で `next@16.2.0` 以上へのアップデートを行うことは推奨されます（緊急性は低）。

### 2.5 総合的に、このリポジトリをイントラネットでホスティングしても安全か？

**判定: 条件付きホスティング可**

静的サイトであり、外部公開もしないため、基本的なリスクは低いです。以下の条件を満たすことで、安全にホスティング可能です。

**【必須条件】**
1.  **NginxでのCSPヘッダー付与:** `security-audit-report.md` の "Additional Findings A1" に従い、特に `script-src` を適切に制限したCSPを設定すること。
2.  **Basic認証の適切な管理:** `.htpasswd` をソースコードに含めず、デプロイプロセスで安全に注入すること。

**【推奨事項】**
1.  **Dockerコンテナの非root化:** インフラセキュリティのベストプラクティスとして実施を推奨。
2.  **HTTPS化:** イントラネットであっても、認証情報（Basic認証ヘッダー）が平文で流れるのを防ぐため、自己署名証明書でも良いのでHTTPS化を検討すること。
